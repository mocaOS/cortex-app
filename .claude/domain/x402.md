# x402 Payments (pay-per-query monetization)

Monetize the retrieval endpoints via [x402 v2](https://github.com/x402-foundation/x402) micropayments. A "monetized public key" is a normal `APIKey` node carrying `price_per_query`: the key stays the identity + collection-scoping mechanism (you can sell retrieval over a subset of the library), x402 adds the payment gate. Free member keys are untouched — both models run side by side on one instance.

## Activation model (env flag → runtime config → verification)

1. **`X402_ENABLED=true`** (env, the only x402 env var) unlocks the feature: the "x402 Payments" section appears on `/admin` and the config API accepts writes. Off = the whole surface is hidden and priced keys 403.
2. **Runtime config** (admin-edited, stored in Neo4j — never env): recipient wallet (`pay_to`, global per instance), facilitator URL, CAIP-2 network, asset (address/name/decimals/EIP-712 version), `max_timeout_seconds`, optional `service_name`, optional facilitator auth headers (encrypted at rest via `ENCRYPTION_KEY`, like git PATs).
3. **Verification** binds usability to correctness: priced keys can only be created — and paid requests only served — while the saved config has passed the verify suite. `verified_hash` = sha256 over the payment-relevant fields (+ decrypted auth headers); any change to those fields silently invalidates verification and the gate fails closed (503) until re-verified.

## Storage (`neo4j_service.py`)

- **`X402Config {id:'x402'}`** singleton — deliberately NOT `SystemMeta`: payment config is instance-operational (like git connections), so it is **excluded from library export and survives System Reset**. Methods: `get_x402_config` / `save_x402_config` / `mark_x402_verified`.
- **`X402Payment`** — one node per settled payment (tx hash, network, payer, `amount_atomic`, endpoint) with `key_id`/`key_name` **denormalized** so earnings survive key deletion. `record_x402_payment` / `get_x402_earnings` (atomic sums; human formatting happens in the service using the configured decimals). Also excluded from export/reset.
- **`APIKey.price_per_query`** — human-unit decimal string (e.g. `"0.05"`); null = free key. Rides every key query; monetized keys mint with the `cortex_pub_` prefix.

## The payment gate (`services/x402_service.py`)

`enforce_x402_payment` — FastAPI dependency on exactly four endpoints: `/api/search`, `/api/ask`, `/api/ask/stream`, `/api/ask/stream/thinking` (declared after `enforce_query_quota`/`enforce_rate_limit`, so quota/rate rejections happen **before** money moves). No-op for free keys/admin. For priced keys, x402 v2 HTTP transport:

1. No `PAYMENT-SIGNATURE` header → **402** + base64 `PAYMENT-REQUIRED` header (requirements built server-side from config + key price; amounts via `Decimal`, never float).
2. Payload present → **never trust the client's `accepted`**: rebuild requirements and compare field-by-field (`accepted_matches_requirements`, addresses case-insensitive) → facilitator `POST /verify` → `POST /settle` → serve.
3. **Settle-before-serve**: EIP-3009 nonce burn makes replays impossible without a server-side nonce store, and SSE responses need headers before the first byte. The `SettlementResponse` is stashed on `request.state.x402_payment_response`; the request-id middleware in `main.py` emits it as the `PAYMENT-RESPONSE` header.
4. Settlement recorded best-effort (`X402Payment`); a bookkeeping failure never fails a paid request.

**Fail-closed everywhere**: facilitator unreachable → 503 + Retry-After (mirrors the auth `service_error` philosophy); config unverified → 503; feature flag off → 403. Paid queries still consume `MAX_QUERIES_PER_MONTH` units (inference cost is real).

**Vendor agnosticism**: the spec's facilitator REST interface (`GET /supported`, `POST /verify`, `POST /settle`) IS the vendor seam — `FacilitatorClient` is a thin httpx client, no vendor SDK, no chain RPC, no signing keys. Static auth headers cover most facilitators; CDP-style dynamic JWT auth would be a pluggable header-provider extension.

## Monetized-key hardening (`auth_service.py`, `main.py`)

Public paid keys are locked down at the API level, not just the UI:

- **Endpoint allowlist** — `MONETIZED_KEY_ALLOWED_PATHS` (the 4 retrieval endpoints, checked in `require_read_permission`). Plain READ would hand out the raw corpus (document content / file download / graph browse) for free next to the paid retrieval. `/api/stats` is deliberately excluded (internal data).
- **price ⊕ MANAGE, both directions** — create/update reject the combination with 422 against the *effective* post-update state (clearing the price with `price_per_query: ""` in the same PATCH re-allows manage). Defense in depth: `validate_api_key` strips MANAGE from any priced key at validation time, so even a hand-edited node never authenticates read-write.
- **Creation gate** — `_validate_monetized_key_price` (main.py): price requires `X402_ENABLED` + verified config, read-only permissions, and a positive decimal within the asset's precision.

## Verify suite & admin endpoints (`main.py`, admin-only)

- `GET/PUT /api/admin/x402/config` — read (secrets masked, carries the env flag for UI gating) / save (PUT 400 while flag off; trailing-slash-normalizes the facilitator URL).
- `POST /api/admin/x402/verify` — check suite: `payto_format` + `asset_format` (EIP-55 checksum via a pure-Python keccak-256, pinned by test vectors; base58/32-byte for Solana), `facilitator_reachable` (`GET /supported`), `scheme_network_supported` (`exact` @ configured network in `kinds`). All pass → `mark_x402_verified(config_hash)`.
- `GET /api/admin/x402/earnings` — instance totals + per-key breakdown, human units.

## Frontend

- `X402Section` (`components/admin/`) on `/admin`, rendered only when the config response says `enabled`: config form (network/USDC presets + custom, inline address format hints), verify-checklist UI, Verified/Unverified status chip, earnings block.
- Key modal: key-type choice (member vs monetized public); monetized forces read-only + price input, disabled until config verified. `ApiKeyCard` shows a price badge.

## Gotchas

- **Deploy composes must map `X402_ENABLED` explicitly.** The Coolify/Dokploy composes enumerate env vars (no `env_file`), so setting the flag in the platform UI does nothing until the compose carries `- X402_ENABLED=${X402_ENABLED:-false}` (added 2026-07-17; symptom: key modal shows the monetized option disabled but the /admin section is missing). `docker-compose.prod.yml` and the dev compose inherit `.env` wholesale and are unaffected.
- CORS `expose_headers` includes `PAYMENT-REQUIRED`/`PAYMENT-RESPONSE` — without it, browser-based agents cannot read the 402 requirements.
- **402 ≠ error in usage analytics**: `APIUsageMiddleware` excludes 402 from `is_error` — a challenge is the first leg of every paid request, so counting it would pin a healthy monetized key's error rate near 50%.
- The x402 config cache is module-global with a 30s TTL (`invalidate_x402_config_cache()` on save/verify; reset per test in conftest, like the auth cache).
- Keccak-256 here is the **pre-SHA3 padding variant** (0x01 domain) — `hashlib.sha3_256` would validate wrong checksums. Test vectors in `test_x402.py` pin it.
- **Paying agents should use `/api/ask/stream`, not `/api/ask`**: the non-streaming endpoint's ~28s `deadline_exceeded` 504 can fire *after* settlement (settle-before-serve) — money for no answer. The stream endpoint has no deadline and carries `PAYMENT-RESPONSE` on its headers before the first byte (live-confirmed 2026-07-17: first real mainnet payment settled, then the non-streaming answer 504'd).
- **`asset_name` is the EIP-712 domain name, not a display label.** It must equal the token contract's `name()` — Circle's USDC differs per deployment ("USD Coin" on Base/Avalanche mainnets, "USDC" on Base Sepolia); a mismatch makes every settlement revert on-chain (live-confirmed 2026-07-17). It's in `_HASHED_FIELDS`, and the admin UI's USDC presets carry the per-network on-chain names.
- Tests: `backend/tests/test_x402.py` (protocol primitives, gate paths with a faked facilitator, hardening, endpoint guards). The facilitator is always mocked — no network in tests.
